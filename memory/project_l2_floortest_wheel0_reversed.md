**2026-07-22: L2 floor drive test RAN (first armed floor test). Two corrections after user physically observed it:**

**CORRECTION 1 — wheel 0 is NOT reversed (earlier FAIL was a false alarm).** User physically confirmed ALL FOUR wheels drive forward. The esc_status reading `[-234, 237, 238, 238]` (wheel 0 negative) is just SIGN CONVENTION — that VESC reports opposite-sign RPM (mirrored motor mount / per-VESC direction config) while the shaft still drives the chassis forward. `l2_test.py`'s check `all(r>=0) or all(r<=0)` is TOO NAIVE for a real skid-steer and wrongly flagged FAIL. **No polarity fault exists. Do NOT chase a wheel-0 fix.** RPM sign in esc_status is NOT a reliable indicator of physical direction on this hardware.

**CORRECTION 2 (the real problem) — l2_test.py SOFTWARE-ARMS, bypassing the RC arm switch.** Line 148 `n.set_armed(True)` sends COMPONENT_ARM_DISARM param1=1, arming the vehicle WITHOUT the user's RC ch5. It then drove /cmd_vel forward 0.2 m/s and the rover headed toward a WALL on its own; user stopped it with the KILL switch (ch8) before impact. UNSAFE for a test tool — the human must be the arming authority.

**PROVEN GOOD (first-ever, armed, on floor):** arm+AutoNav engage (nav_state=23); **watchdog** zeros motors within 2s of /cmd_vel stop; **kill switch (ch8) works ARMED inside AutoNav** (todos #19 — now confirmed live); auto-disarm+Hold on exit. Bridge stopped after → safe.

**FIX PLANNED for l2_test.py (proposed to user, awaiting go-ahead):** remove software-arm; step 3 instead WAITS for the operator to arm via RC (poll until arming_state==ARMED, else Ctrl-C abort) — only drives if the human armed it. Also reduce MOTION_SECONDS so first floor run is a nudge, not 0.6 m. Kill/disarm stops everything at any point.
